# 管理页面新增开发流程模板

这份文档是给“新增一个后台管理功能页面”用的标准步骤说明。

适用场景：

- 新增一个列表管理页面
- 新增一个配置页面
- 新增一个带增删改查的业务页面
- 前后端需要一起推进的功能

目标：

- 让自己以后不容易忘
- 每次都按同一套步骤推进
- 降低漏做权限、接口、联调、自测的风险

---

## 1. 使用方式

每次新增一个功能时，先复制这份文档里的“开发记录模板”到你自己的临时笔记里。

然后严格按顺序执行：

1. 先确认需求
2. 再做后端接口
3. 再做前端页面
4. 再做权限接入
5. 再联调
6. 最后自测和提交

建议不要跳步骤。

---

## 2. 新增页面前先确认的内容

在开始写代码前，先把下面这些信息写清楚。

### 2.1 基础信息

- 功能名称：
- 所属模块：
- 菜单名称：
- 路由地址：
- 权限前缀：

示例：

- 功能名称：设备管理
- 所属模块：系统管理
- 菜单名称：设备管理
- 路由地址：`/system/device`
- 权限前缀：`system:device`

### 2.2 页面功能范围

确认本页需要哪些功能：

- 列表查询
- 分页
- 新增
- 编辑
- 删除
- 批量删除
- 查看详情
- 导出
- 启用/禁用
- 分配权限
- 其他特殊功能

### 2.3 字段清单

把字段先列出来，不要边写边猜。

| 字段名 | 中文名 | 类型 | 查询条件 | 列表展示 | 表单编辑 | 备注 |
|---|---|---|---|---|---|---|
| id | 主键 | number | 否 | 否/是 | 否 | 主键 |
| name | 名称 | string | 是 | 是 | 是 | 必填 |
| status | 状态 | number | 是 | 是 | 是 | 0/1 |
| remark | 备注 | string | 否 | 是 | 是 | 非必填 |

### 2.4 权限点清单

权限点建议统一命名为：

`模块:资源:动作`

示例：

- `system:device:list`
- `system:device:add`
- `system:device:edit`
- `system:device:delete`
- `system:device:view`
- `system:device:export`

---

## 3. 标准开发顺序

以后每新增一个管理页，固定按这个顺序走。

1. 找一个最像的旧页面参考
2. 确认需求和字段
3. 确认后端表结构和接口
4. 确认权限点
5. 先完成后端接口
6. 再完成前端页面骨架
7. 接列表查询
8. 接新增编辑
9. 接删除和批量删除
10. 接按钮权限
11. 联调和异常处理
12. 自测
13. 提交代码

不要一上来就先堆前端页面，否则最后很容易发现接口不匹配。

---

## 4. 后端开发步骤

这一段是你做后端时按顺序执行的。

### Step 1：确认数据表

先确认：

- 有没有现成表可以复用
- 是否需要新建表
- 表字段是否齐全
- 是否需要状态字段
- 是否需要创建时间、更新时间、创建人、更新人

常见字段建议：

- `id`
- `name`
- `code`
- `status`
- `remark`
- `create_time`
- `update_time`
- `create_by`
- `update_by`

### Step 2：确认接口清单

一个标准管理页通常至少有这些接口：

- 分页列表接口
- 详情接口
- 新增接口
- 编辑接口
- 删除接口
- 批量删除接口

可选接口：

- 导出接口
- 修改状态接口
- 下拉选项接口

示例：

- `GET /system/device/page`
- `GET /system/device/{id}`
- `POST /system/device`
- `PUT /system/device`
- `DELETE /system/device/{id}`
- `DELETE /system/device/batch`

### Step 3：统一返回结构

分页接口最好统一返回：

```json
{
  "list": [],
  "total": 0
}
```

详情接口最好统一返回对象。

这样前端复用会轻松很多。

### Step 4：完成权限控制

后端接口必须做权限校验，不能只靠前端隐藏按钮。

至少确认：

- 列表接口需要 `list`
- 新增接口需要 `add`
- 编辑接口需要 `edit`
- 删除接口需要 `delete`
- 查看接口需要 `view`

### Step 5：完成参数校验

后端需要校验：

- 必填字段
- 长度限制
- 唯一性
- 业务规则

例如：

- 名称不能为空
- 编码不能重复
- 状态值只能是 0 或 1

### Step 6：先用接口工具自测

在前端联调前，先自己测一遍接口。

至少测这些：

- 查询正常
- 空条件查询正常
- 新增正常
- 编辑正常
- 删除正常
- 错误参数会报错
- 无权限时会被拒绝

## 5. 前端开发步骤

这一段是你做前端时按顺序执行的。

### Step 1：先找一个最像的页面复制

不要从零写。

优先找相似页面：

- 一样是列表管理页
- 一样有查询条件
- 一样有弹窗表单
- 一样有按钮权限

复制后重点改下面几类内容：

- 页面名称
- 路由
- 权限编码
- API 名称
- 字段名
- 表格列
- 表单字段

### Step 2：先建页面目录

推荐目录：

```text
src/api/system/device.ts
src/views/system/device/index.vue
src/views/system/device/components/DeviceDialog.vue
```

如果有详情弹窗，再加：

```text
src/views/system/device/components/DeviceDetail.vue
```

### Step 3：先把路由和菜单接好

确认这些内容：

- 路由 path
- 路由 name
- 菜单 title
- 组件路径
- 页面权限标识

如果菜单是后端动态配置，就确认：

- 前端组件路径可匹配
- 后端菜单权限编码正确

### Step 4：先写 API 文件

不要把请求直接散写在页面里。

标准接口通常包括：

- `getPage`
- `getDetail`
- `add`
- `update`
- `delete`
- `batchDelete`

示例命名：

- `getDevicePage`
- `getDeviceDetail`
- `addDevice`
- `updateDevice`
- `deleteDevice`
- `batchDeleteDevice`

### Step 5：先搭列表页骨架

页面结构建议固定为：

1. 查询区
2. 工具栏按钮区
3. 表格区
4. 分页区
5. 新增/编辑弹窗

不要先做复杂样式，先把流程跑通。

### Step 6：接列表查询

先实现这些内容：

- 查询参数对象
- 表格数据
- 总数
- loading
- `getList`
- `handleQuery`
- `resetQuery`

建议顺序：

1. 先让默认列表显示出来
2. 再做查询条件
3. 再做分页

### Step 7：接新增/编辑弹窗

新增编辑最好共用一个弹窗组件。

需要准备：

- `mode` 区分新增和编辑
- `visible` 控制弹窗显示
- `formData` 表单对象
- `rules` 校验规则
- `handleSubmit` 提交方法

建议流程：

- 新增时清空表单
- 编辑时根据 id 查详情回填
- 提交成功后关闭弹窗并刷新列表

### Step 8：接删除和批量删除

删除标准流程：

1. 点击删除
2. 二次确认
3. 调接口
4. 成功提示
5. 刷新列表

批量删除要注意：

- 记录选中项 ids
- 未选择时按钮禁用
- 删除后清空选中状态

### Step 9：接详情功能

如果需要只读查看，建议单独做详情弹窗，不要混在编辑表单里。

适合详情展示的内容：

- 全部字段
- 图片
- 富文本
- 操作记录

### Step 10：接字典和枚举值

常见内容：

- 状态
- 类型
- 分类
- 标签

要求：

- 查询区显示一致
- 表格展示一致
- 表单提交值一致

避免一个页面里同一个字段出现三套不同文案。

### Step 11：接按钮权限

这是管理后台一定不能漏的。

常见按钮权限：

- 新增：`xxx:add`
- 编辑：`xxx:edit`
- 删除：`xxx:delete`
- 查看：`xxx:view`
- 导出：`xxx:export`

典型控制方式：

```vue
<el-button v-if="hasPerm('system:device:add')">新增</el-button>
```

要求：

- 没权限时按钮不显示
- 页面访问权限单独控制
- 前端按钮权限和后端接口权限都要做

### Step 12：补异常处理

至少处理这些：

- 列表加载中状态
- 提交按钮 loading
- 接口失败提示
- 空数据状态
- 删除最后一条后的分页回退

---

## 6. 联调步骤

前后端都完成后，再按这个顺序联调。

### Step 1：先调列表

确认：

- 查询参数名是否一致
- 分页参数名是否一致
- 返回结构是否一致
- 时间字段格式是否一致

### Step 2：再调新增

确认：

- 字段提交名是否一致
- 必填校验是否一致
- 提交成功返回是否一致

### Step 3：再调编辑

确认：

- 详情接口返回字段能否正常回填
- 更新提交字段是否完整

### Step 4：再调删除

确认：

- 单删正常
- 批量删除正常
- 无权限角色删除会失败

### Step 5：再调权限

至少用两类角色测试：

- 管理员角色
- 普通角色/只读角色

确认：

- 不同角色看到的按钮不一样
- 无权限时接口不能调用成功

## 7. 提交前自测清单

每次开发完成后，至少对一遍下面这份清单。

### 7.1 页面访问

- [ ] 路由能正常打开
- [ ] 菜单能正常显示
- [ ] 页面刷新不报错
- [ ] 页面标题正确

### 7.2 列表功能

- [ ] 默认能查到数据
- [ ] 查询条件有效
- [ ] 重置功能正常
- [ ] 分页切换正常
- [ ] pageSize 切换正常
- [ ] 空数据展示正常

### 7.3 新增编辑

- [ ] 新增弹窗能打开
- [ ] 编辑弹窗能回填
- [ ] 表单校验正常
- [ ] 提交按钮有 loading
- [ ] 提交成功后会刷新列表

### 7.4 删除功能

- [ ] 单条删除正常
- [ ] 批量删除正常
- [ ] 删除后列表刷新正常
- [ ] 删除最后一条时页码处理正常

### 7.5 权限功能

- [ ] 无页面权限时不可访问
- [ ] 无新增权限时不显示新增按钮
- [ ] 无编辑权限时不显示编辑按钮
- [ ] 无删除权限时不显示删除按钮
- [ ] 无接口权限时后端会拒绝

### 7.6 展示效果

- [ ] 字段展示正确
- [ ] 状态映射正确
- [ ] 时间格式正确
- [ ] 长文本不撑坏布局
- [ ] 小屏下页面不明显错位

### 7.7 异常处理

- [ ] 接口报错有提示
- [ ] 网络慢时有 loading
- [ ] 空值不会导致页面报错

---

## 8. 提交代码前的最后检查

提交前再看一遍：

1. 是否有调试代码没删
2. 是否有 `console.log`
3. 是否有无用注释
4. 是否有没用到的 import
5. 是否有权限编码写错
6. 是否有接口地址写错
7. 是否有表单字段名和接口字段名不一致

如果项目要求，还要执行：

1. 前端 lint
2. 前端 build
3. 后端单测或最小验证

## 10. 推荐执行原则

最后记住这几条，能帮你减少很多返工。

1. 先做接口，再做页面
2. 先跑通列表，再做弹窗
3. 先保证流程通，再优化样式
4. 按钮权限和接口权限必须同时做
5. 每次都找一个旧页面复用，不要重头写
6. 每次开发完成都按清单自测

如果以后要新增的不是普通 CRUD 页面，而是：

- 树形页面
- 主子表页面
- 多 tab 详情页
- 带文件上传页面

也建议先基于这份流程，再补充对应的专项步骤。
